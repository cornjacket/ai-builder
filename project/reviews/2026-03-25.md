# Session Review — 2026-03-25

Scope: this session only. Critique by Claude, responses by David.

---

## 1. Designed in chat instead of in a document

**Critique:** The doc/review/update pipeline design happened entirely in chat.
Key decisions had to be reconstructed into a brainstorm file after the fact,
with inevitable gaps and lost nuance.

**Response:** Good lesson. CLAUDE.md updated with a rule: when a brainstorm
begins, immediately create `sandbox/brainstorm-{subject}.md` and write to it
in real time. See `lessons/040-brainstorm-to-file-not-chat.md`.

## 2. Diagnosed problem before reading the code

**Critique:** The existing brainstorm described "unbounded growth" as the handoff
history risk. The actual bug was the opposite — complete erasure on every sibling
transition. A fix was being designed against a problem that hadn't been confirmed
in the code. The correct move was to read the actual orchestrator code and trace
the frame stack paths before reaching for a solution.

**Response:** The misdiagnosis originated with the AI, not the user. The task was
named and framed around "unbounded growth" by a prior Claude session. The correct
diagnosis only emerged when David asked Claude to walk through the actual
platform-monolith flow — at which point the always-clears bug became obvious from
the concrete example. The diagnosis was user-directed, not AI-initiated.

A second failure: given that level-scoped frame stack push/pop is a well-understood
pattern, Claude should have derived the correct two-frame algorithm from first
principles once the problem was properly described. Instead it took David explicitly
proposing the design before Claude engaged with it properly. That is a failure of
applying domain knowledge, not just a diagnosis miss.

**Revised framing:** This critique applies to Claude's prior session (misdiagnosis)
and this session (failure to derive the correct algorithm independently). Not a
user mistake.

## 2. Planning-to-shipping ratio is badly skewed / explicitly deferred starting work

**Critique:** The session produced 18 subtask specs, 3 backlog tasks, 1 reorder
script, and 0 lines of orchestrator code. The generic framing: too much planning,
not enough shipping. Also: said "before we begin the first task I want to discuss
a new task" and never came back to implementation.

**Response:** Disagree. Poor planning leads to incomplete designs. The effort for
the pipeline redesign is well worth it. The original pipeline's communication
architecture problems are direct evidence of building incrementally without a
design — it now requires an 18-subtask restructure to fix. Capturing key insights
at the moment they occur takes highest priority; implementation follows naturally
once the design is solid. The brainstorming of the documenter and redesigned
pipeline led to the realisation that the pipeline has multiple potential use cases —
that insight needed to be captured immediately. The pipeline restructure is now
well documented. Implementation is now just saying go. Also: using AI has a
tendency to encourage thinking-with-the-AI since it's good at anticipating system
patterns, but experimentation is how the pipeline-task system evolved and some
ideas would never have happened without it.

**Claude's revised position:** Both critiques retracted. The repurposing discussion
was new work that needed brainstorming before implementation could begin — not a
deferral. Capturing pipeline use-case insights at the moment they occurred was the
correct priority. The only valid critique in this space remains item 1 — reading
code before writing a design doc.

## 3. Delegating to Claude without tight enough specs, then absorbing the correction cost

**Critique:** "Fill in tasks 0000-0013" with only a brainstorm as input produced
output that needed corrections — specifically the implementation sequence numbers
in the parent README. The rework negated the efficiency of batch delegation.

**Response:** The sequence number correction was not a user mistake. It was a gap
in the task system documentation — the convention that `NNNN` defines implementation
order was never explicitly written down anywhere. The AI didn't know to apply it
when filling in the parent README sequence line. The fix: document the convention
explicitly in CLAUDE.md, `target/init-claude-md.sh`, and `project/README.md` so
it is unambiguous going forward.

**Claude's revised position:** Retracted. Not a delegation spec problem — a
documentation gap. Closed by the convention update.

## 4. The directory-name ordering convention is a self-inflicted maintenance burden

**Critique:** Embedding implementation order in directory names means every plan
change requires a rename script. `reorder-subtasks.py` was built to solve a
problem caused by the naming convention itself.

**Response:** Disagree strongly. The visual ordering prior to adding the numbering
was a nightmare to track in real time during active implementation. That one
feature is well worth the cost. The fact that the script wasn't needed for so
long shows it wasn't much of a burden — it was built exactly when needed, not
speculatively. That's good engineering judgment.

**Claude's revised position:** Retracted. The convention solved a real pain point;
the tooling cost was deferred until necessary and proportionate to the value gained.
