## Description

Design a DOCUMENTER agent role for the ai-builder orchestrator. The DOCUMENTER
understands and maintains the **index of index of READMEs** pattern in a target
application, keeping documentation in sync with the codebase as it evolves.

**What the DOCUMENTER does:**
- Traverses the target application's directory tree
- Ensures every significant component directory has a `README.md` that describes
  what it does, its inputs/outputs, and links to its sub-components
- Maintains a root-level `docs/INDEX.md` (or equivalent) that links to every
  layer's README, forming a complete navigable map of the system
- When new code is generated by IMPLEMENTOR, the DOCUMENTER updates the relevant
  README(s) and index entries to reflect the change
- Ensures documentation is written for both human and AI audiences

**Index of index of READMEs pattern:**
Each directory has a README describing that layer. The root index links to all
top-level READMEs. Each top-level README links to its children. This mirrors the
directory structure, stays co-located with the code, and requires no special
tooling to traverse — any agent can navigate it with standard file reads.

**Design questions to resolve:**
- When in the pipeline does DOCUMENTER run? (after TESTER signs off, before
  PM marks complete? or as a parallel step?)
- What is the minimum viable README for a new component?
- How does DOCUMENTER handle deleted or moved components?
- Should DOCUMENTER verify that existing docs are still accurate, or only
  add/update?
- What is the handoff protocol between IMPLEMENTOR and DOCUMENTER?
